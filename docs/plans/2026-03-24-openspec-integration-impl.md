# OpenSpec Integration Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Install and initialize openspec in the orcai repo, wire up Claude Code slash commands, and validate the dev workflow.

**Architecture:** Track 1 only — this plan covers the dev workflow setup. Track 2 (Tier 2 CLI wrapper plugin) is a separate plan, written after CliAdapter is built.

**Tech Stack:** Node.js/npm, openspec CLI (`@fission-ai/openspec`), Claude Code slash commands

---

### Task 1: Install openspec globally

**Files:**
- No files changed — global npm install

**Step 1: Install openspec**

```bash
npm install -g @fission-ai/openspec@latest
```

Expected: installs without error, `openspec` binary available in PATH.

**Step 2: Verify installation**

```bash
openspec --version
```

Expected: prints a version string (e.g. `0.x.x`).

---

### Task 2: Initialize openspec in the orcai repo

**Files:**
- Create: `.openspec/` (config dir, generated by init)
- Create: `openspec/` (proposals dir, generated by init)

**Step 1: Run init**

```bash
cd /Users/stokes/Projects/orcai
openspec init
```

Expected: creates `.openspec/` and `openspec/` directories, generates Claude Code guidance files.

**Step 2: Inspect generated files**

```bash
ls .openspec/
ls openspec/
```

Note which files were created — some will be committed, others may be gitignored.

**Step 3: Update .gitignore if needed**

Check if a `.gitignore` exists and whether openspec generated any local-only files that should be excluded. Add entries as needed. Do not gitignore `.openspec/` config itself — it should be committed so the team gets the Claude Code slash commands.

**Step 4: Commit generated files**

```bash
git add .openspec/ openspec/ .gitignore
git commit -m "chore: initialize openspec for spec-driven development"
```

---

### Task 3: Validate Claude Code slash commands

**Files:**
- No code changes — verification only

**Step 1: Check that slash commands are wired**

In a Claude Code session in the orcai directory, type:

```
/opsx:propose add a settings panel to the prompt builder sidebar
```

Expected: openspec generates a proposal folder under `openspec/changes/active/` containing `proposal.md`, `design.md`, `tasks.md`, and `specs/`.

**Step 2: Inspect the generated proposal**

```bash
ls openspec/changes/active/
cat openspec/changes/active/*/proposal.md
```

Verify the proposal has meaningful content — rationale, scope, design approach.

**Step 3: Archive the test proposal (do not apply it)**

```
/opsx:archive
```

Or delete the test proposal folder manually — this was just to validate the workflow.

---

### Task 4: Generate the first real proposal

**Files:**
- Creates: `openspec/changes/active/YYYY-MM-DD-<feature>/` with full proposal structure

**Step 1: Pick the next feature to build**

Based on the existing plugin system design (`docs/plans/2026-03-24-plugin-system-design.md`), the natural next feature is the `CliAdapter` (Tier 2 plugin wrapper) since it unblocks the openspec plugin integration.

**Step 2: Generate a proposal**

In a Claude Code session:

```
/opsx:propose implement the Tier 2 CliAdapter so any CLI tool in PATH is automatically wrapped as an orcai plugin with stdin/stdout JSON envelope communication and optional sidecar YAML schema declaration
```

**Step 3: Review the proposal**

```bash
cat openspec/changes/active/*/proposal.md
cat openspec/changes/active/*/design.md
cat openspec/changes/active/*/tasks.md
```

Review for accuracy against the plugin system design. Edit any tasks that conflict with the existing architecture decisions.

**Step 4: Commit the proposal**

```bash
git add openspec/changes/active/
git commit -m "spec: propose CliAdapter (Tier 2 plugin wrapper)"
```

## Track 2 Note

The Tier 2 CLI wrapper for openspec (exposing it as a prompt builder pipeline step) is a separate implementation plan. Write it after:
1. `internal/plugin/` CliAdapter is implemented
2. `internal/pipeline/` interpreter is implemented
3. The sidecar YAML loading is in place

At that point, the entire integration reduces to writing `~/.config/orcai/wrappers/openspec.yaml` — no code changes required.
