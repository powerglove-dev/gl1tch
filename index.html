ABOUT

// orcai — abbs //

Browser-based AI tools charge you for every token spent staring at a loading spinner. orcai runs in your terminal, next to your editor and shell, using the models you already have. Local inference, zero cloud markup. Your API spend goes to actual work — not UI overhead.

[ COST & EFFICIENCY ]
LOCAL MODELS
Run Ollama on-device. No API call, no bill.
Pull qwen2.5-coder once, run it forever.
TYPED PIPELINES
No glue code means no wasted tokens fixing bad output formats.
Steps get what they expect. First time, every time.
BACKGROUND CRON
Agents run while you sleep. Wake up to results,
not a context window full of back-and-forth prompting.

BRAIN is context injection. Point it at docs, files, URLs, or code. Every agent run gets that context loaded automatically — your agents know your codebase, your standards, your project layout. They stop asking questions you've already answered elsewhere.

[ BRAIN, PLUGINS & PRODUCTIVITY ]
BRAIN CONTEXT
Docs, files, URLs loaded once, reused on every run.
No copy-pasting into chat windows.
TMUX SESSIONS
Each agent gets its own workspace. Switch mid-pipeline,
kill when done. Terminal stays yours.
PLUGIN SYSTEM
YAML file pointing at any CLI: rg, jq, ffmpeg, your scripts.
Need more? Implement the gRPC interface in Go.
INBOX + FEED
Every agent run lands in a scrollable inbox.
Review outputs, replay steps, full audit trail.

GETTING STARTED

// from zero to running agent //

01 · INSTALL
go install github.com/adam-stokes/orcai@latest

orcai lands in $GOPATH/bin — make sure it’s on your $PATH.

02 · PLUGINS

Plugins extend orcai with CLI tool adapters. Place wrapper YAML files in ~/.config/orcai/wrappers/.

ollama — local models

# ~/.config/orcai/wrappers/ollama.yaml
name: ollama
type: sidecar
command: ollama
args: ["run", "{{model}}", "{{prompt}}"]

claude — cloud AI

# ~/.config/orcai/wrappers/claude.yaml
name: claude
type: sidecar
command: claude
args: ["-p", "{{prompt}}", "--model", "{{model}}"]

gh — GitHub CLI

# ~/.config/orcai/wrappers/gh.yaml
name: gh
type: sidecar
command: gh
args: ["{{args}}"]

jq — JSON processor

# ~/.config/orcai/wrappers/jq.yaml
name: jq
type: sidecar
command: jq
args: ["{{filter}}"]
03 · OLLAMA
OLLAMA SUPERCHARGER

Run fully local AI — no API key, no cloud, no cost.
Ollama supercharges orcai with private, fast, offline models.

ollama pull qwen2.5-coder:latest
ollama pull llama3.2
ollama serve

qwen2.5-coder is excellent for code tasks. llama3.2 is great for general reasoning and brain context pipelines.

04 · FIRST PIPELINE

Launch orcai and run the brain Ollama E2E pipeline — it tests the brain feedback loop using a local model. 

orcai

In the Switchboard → PIPELINES panel, select brain-ollama-e2e and press Enter to launch. Or run directly: 

orcai pipeline run brain-ollama-e2e

Step 1 (write_brain) asks qwen2.5 to summarize itself and saves the result as a brain note. Step 2 (use_brain) asks qwen2.5 to reference that note — verifying the brain context feedback system works. 

[brain-ollama-e2e] step-write: running...
[brain-ollama-e2e] step-write: done (brain note saved)
[brain-ollama-e2e] step-read:  running (brain context injected)...
[brain-ollama-e2e] step-read:  done
pipeline complete ✓
05 · FIRST AGENT

Launch an interactive agent session directly from the Switchboard.

  1. Press Enter on the AGENT RUNNER panel in the Switchboard
  2. Select your provider (e.g., Ollama)
  3. Tab to MODEL → select llama3.2 (Ollama) or any available model
  4. Tab to PROMPT → type your task
  5. Optionally: Tab to [ ] use brain context and press Space to enable brain injection
  6. Press Ctrl+S to submit

PLUGINS

// plugin architecture //

Plugins are YAML sidecar files in ~/.config/orcai/wrappers/. Two kinds: agent plugins drive AI providers, tool plugins wrap CLI tools. No registry, no daemon, no installation ceremony beyond dropping a file.

[ BUNDLED PLUGINS ]
ollama
[agent] Local models via the Ollama daemon. Models discovered at runtime.
orcai-ollama binary + ollama.yaml
claude
[agent] Claude AI via the claude CLI. Opus, Sonnet, Haiku.
orcai-claude binary + claude.yaml
gemini
[agent] Google Gemini via the gemini CLI. Flash and Pro models.
orcai-gemini binary + gemini.yaml
github-copilot
[agent] GitHub Copilot via gh extension.
orcai-github-copilot + yaml
opencode
[agent] OpenCode AI assistant.
orcai-opencode binary + yaml
gh
[tool] GitHub CLI wrapper. Pass any gh subcommand via vars.args.
gh.yaml only (no binary)
jq
[tool] Apply a jq filter to JSON input. Filter via vars.filter.
jq.yaml only (no binary)

Agent plugins declare their models. Tool plugins wrap a shell command. Drop the YAML in ~/.config/orcai/wrappers/ and orcai discovers it on next launch.

// agent sidecar (kind: agent)

name: ollama
description: "Run local AI models via the Ollama daemon"
kind: agent
command: orcai-ollama
models:
  - id: llama3.2:latest
    label: "Llama 3.2"
  - id: qwen2.5-coder:latest
    label: "Qwen 2.5 Coder"

// tool sidecar (kind: tool)

name: jq
description: "Apply a jq filter to JSON input"
kind: tool
command: sh
args:
  - "-c"
  - 'jq "${ORCAI_FILTER:-.}"'
[ GRPC SIDECAR — FULL CONTROL ]
interface
Implement the Plugin gRPC interface in Go for full control over execution, streaming, and output types.
discovery
Binary on PATH + YAML descriptor. orcai spawns and communicates via gRPC — no coupling to orcai internals.
type Plugin interface {
    Describe() PluginInfo
    Execute(ctx context.Context, input Input) (Output, error)
}

// runnable pipeline examples //
  
 
LAB A · BRAIN FEEDBACK LOOP (OLLAMA E2E)
 

Two-step brain test — step 1 writes a note, step 2 reads it back.
      Validates the brain context feedback system with a local model.

 
// prerequisites
 
 ► Ollama running with qwen2.5:latest pulled

 
// yaml
 
name: brain-ollama-e2e
version: "1"

steps:
  - id: step-write
    plugin: ollama
    model: qwen2.5:latest
    write_brain: true
    prompt: |
      Summarize yourself in one sentence. Write the result
      as a brain note starting with: "INSIGHT:"

  - id: step-read
    plugin: ollama
    model: qwen2.5:latest
    use_brain: true
    prompt: |
      Reference the INSIGHT from the previous step.
      What did it say about itself?
 
// run
 
orcai pipeline run brain-ollama-e2e

LAB C · LOCAL CODE REVIEW WITH OLLAMA
 

Fetches the diff from the latest open PR in any repo, then asks a local
qwen2.5 model for a concise code review. No API key needed.

 
// prerequisites
 
 ► gh CLI authenticated
 ► gh sidecar configured
 ► jq sidecar configured
 ► Ollama with qwen2.5:latest 
 
// yaml
 
name: local-code-review
version: "1"

steps:
  - id: fetch-latest-pr
    plugin: gh
    vars:
      args: "pr list --repo <owner>/<repo> --json number,title --limit 1"

  - id: extract-pr-number
    plugin: jq
    vars:
      filter: ".[0].number"

  - id: fetch-diff
    plugin: gh
    vars:
      args: "pr diff {{steps.extract-pr-number.output}}"

  - id: review
    plugin: ollama
    model: qwen2.5:latest
    prompt: |
      Review this pull request diff concisely.
      Focus on correctness, clarity, and potential bugs.
      Diff: {{steps.fetch-diff.output}}
 
// run
 
orcai pipeline run local-code-review

LAB E · GITHUB ACTIVITY DIGEST
 

Pulls issues and PRs from a repo, enriches with local analysis, then produces a
      structured JSON digest with summary totals and action items.

 
// prerequisites
 
 ► gh CLI authenticated
 ► gh/jq sidecars configured
 ► Ollama with qwen2.5:latest
 ► claude CLI on PATH

 
// yaml
 
name: github-activity-digest
version: "1"

steps:
  - id: fetch-issues
    plugin: gh
    vars:
      args: "issue list --repo <owner>/<repo> --json number,title,state --limit 20"

  - id: fetch-prs
    plugin: gh
    vars:
      args: "pr list --repo <owner>/<repo> --json number,title,state --limit 10"

  - id: local-analysis
    plugin: ollama
    model: qwen2.5:latest
    prompt: |
      Analyze these GitHub issues and PRs. Identify themes and blockers.
      Issues: {{steps.fetch-issues.output}}
      PRs: {{steps.fetch-prs.output}}

  - id: structured-digest
    executor: claude
    model: claude-haiku-4-5-20251001
    prompt: |
      Based on this analysis, produce a JSON digest with:
      - total_issues, total_prs, action_items array
      Analysis: {{steps.local-analysis.output}}
 
// run
 
orcai pipeline run github-activity-digest

DOCS
 
 
 
// the pipeline memory system //
  
 

Brain fields give your pipelines persistent, accumulated context across steps.
      A step can write insights to the brain store and later steps can
read from it — turning a linear pipeline into a knowledge-building loop.

  
use_brain
 
 use_brain: true  — type: bool  |  default: false
 
When set on a pipeline or step, injects accumulated brain context into the
        step's prompt preamble as a <brain> XML block.
        The AI sees previous insights from earlier steps (or past runs if brain notes
        were saved). Set at pipeline level to enable for all steps, or override per-step.
 
  
write_brain
 
 write_brain: true  — type: bool  |  default: false
 
When set on a step, the step's output is appended to the brain context store
        for the current run. Subsequent steps with use_brain: true will see this
        output in their preamble. Multiple write_brain steps accumulate —
        context is concatenated in order.
 
  
step-level override
 
 use_brain: false  — per-step suppression
 
Even when use_brain: true is set at the pipeline level, you can suppress
        injection for a specific step with use_brain: false.
        Useful for data-fetch steps where brain context wastes tokens and adds noise.
 
 
  
EXAMPLES
 
// annotated YAML examples //
  
 >>  Example 1: Brain Feedback Loop — two steps, first writes, second reads

 
name: brain-feedback-loop
version: "1"

steps:
  # Step 1: local model writes an insight to brain
  - id: analyze
    plugin: ollama
    model: qwen2.5-coder:latest
    write_brain: true          # ← output saved to brain context
    prompt: |
      Analyze the code in ./src and write one key insight
      about its architecture. Start with "INSIGHT:"

  # Step 2: cloud model uses that insight
  - id: summarize
    executor: claude
    model: claude-haiku-4-5-20251001
    use_brain: true            # ← receives step 1's INSIGHT in preamble
    prompt: |
      Based on the architecture insight above, suggest
      one high-impact improvement.
  
 >>  Example 2: Selective brain injection — pipeline-level on, one step suppressed

 
name: selective-brain
version: "1"
use_brain: true               # ← all steps receive brain context by default

steps:
  - id: fetch-data
    plugin: gh
    use_brain: false           # ← suppressed: no brain context for data fetch
    vars:
      args: "issue list --json number,title --limit 10"

  - id: analyze
    executor: claude
    model: claude-haiku-4-5-20251001
    # use_brain: true inherited from pipeline level
    prompt: |
      Given the brain context above and these issues:
      {{steps.fetch-data.output}}
      What patterns do you see?
  
BRAIN TIPS
 
// making your AI workspace smarter //
 
 
 [1] 
 Start writes early, reads late —
        Put write_brain: true on early exploratory steps (local/fast models),
        then use use_brain: true on later synthesis steps (cloud/expensive models).
        You get deep context without burning tokens on every step.

 
 
 [2] 
 Suppress on pure data-fetch steps —
        If a step just fetches JSON from an API or runs gh pr list,
        add use_brain: false. Brain context won't help and
        wastes the context window.

 
 
 [3] 
 Name your insights explicitly —
        Prompt your write_brain steps to start their output with
“INSIGHT:” or “FINDING:”
so the brain context is scannable. The AI reading it later will extract the signal faster.

 
 
 [4] 
 Chain local → cloud for cost control —
        Use a cheap local Ollama model (qwen2.5-coder, llama3.2)
        to write brain notes, then a cloud model to synthesize. You get the best of both worlds:
        local privacy + cloud reasoning.

 
 
 [5] 
 Brain context accumulates within a run —
        Multiple write_brain steps in the same pipeline run all accumulate
        into the same context block. Step 3 sees what steps 1 and 2 wrote. Design your pipeline
        like a conversation that builds knowledge progressively.

 
 
  
PIPELINE YAML REFERENCE
 
// compact field reference //
 
  name                             string  Pipeline name, used in orcai pipeline run <name>
  version                          string  Schema version, currently "1"
  use_brain                        bool    Inject brain context into all steps (default: false)
  write_brain                      bool    Append pipeline output to brain store (default: false)
  steps[].id                       string  Unique step identifier, used in {{steps.<id>.output}}
  steps[].executor / plugin        string  Agent provider (claude, ollama, gh, jq, …)
  steps[].model                    string  Model ID (e.g. claude-haiku-4-5-20251001, qwen2.5:latest)
  steps[].prompt                   string  Prompt text; supports {{steps.<id>.output}} interpolation
  steps[].use_brain                bool    Per-step override of pipeline-level use_brain
  steps[].write_brain              bool    Per-step: save this step's output to brain context
