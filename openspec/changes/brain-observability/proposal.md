## Why

The brain produces answers but does not show its work or its trend. The user has no honest way to tell whether the brain is getting better over time, no calibration check on its confidence claims, and no way to tune it without dropping into config files. Once `glitch-research-loop` lands, the brain will be emitting per-signal score events for every answer — that data is exactly what's needed to plot improvement, but only if we choose the *right* metrics, store them coherently, and surface them in chat where the user actually looks.

## What Changes

- Define a small set of **brain quality metrics** computed from the events emitted by the research loop and the chat session: `accept_rate`, `confidence_calibration` (Brier score), `retrieval_precision`, `iteration_count`, `escalation_rate`. No vanity numbers.
- Add a **brain stats engine** in `internal/brainaudit` (or a sibling) that consumes raw events from the brain event store and produces rolling time-series for each metric.
- Add a **`/brain stats`** slash command (depends on `chat-first-ui`) that returns one or more `score_card` widget messages — current value plus a sparkline of the historical series.
- Add a **`/brain config`** widget (depends on `chat-first-ui`) where every research-loop and brain knob (confidence threshold, budgets, enabled researchers, escalation policy) can be edited with action buttons. Every edit emits a `brain_config_change` event so we can correlate config changes with metric movement.
- Add **next-prompt chips**: every assistant answer is followed by an `action_chips` widget containing 2–3 suggested follow-up prompts, generated by a cheap second pass of the same local model that wrote the answer (no extra round trip to the loop).
- Add an **accept/reject feedback signal**: tiny thumbs-up / thumbs-down chips on assistant answers, plus an implicit "ran a contradicting follow-up" detector. Both feed `brain_feedback` events used by the scorer in `glitch-research-loop` to learn weights later.
- Define **retention rules** for brain events: full payloads kept for 30 days, score breakdowns kept indefinitely (they're tiny), config-change events kept indefinitely.

## Capabilities

### New Capabilities
- `brain-quality-metrics`: the definitions of `accept_rate`, `confidence_calibration`, `retrieval_precision`, `iteration_count`, `escalation_rate`, and how each is computed from raw events.
- `brain-stats-engine`: the rolling time-series computation, storage, and query layer for the metrics.
- `brain-config-in-chat`: the `/brain config` slash command, the `widget_card` form, and the `brain_config_change` event emitted on every edit.
- `assistant-next-prompt-chips`: the cheap follow-up generator that produces 2–3 next-prompt suggestions per assistant answer.
- `brain-feedback-signal`: explicit thumbs feedback and implicit contradicting-follow-up detection, both emitted as `brain_feedback` events.

### Modified Capabilities
_None at the spec level. This change consumes events emitted by `glitch-research-loop` and renders into widgets defined by `chat-first-ui`; neither of those needs requirement changes to support this work._

## Impact

- **Depends on**: `glitch-research-loop` (for the per-signal score events) and `chat-first-ui` (for the `score_card`, `widget_card`, and `action_chips` payload types). This change is the third in the dependency chain.
- **Touched packages**: `internal/brainaudit` (stats engine), `internal/brain` (new event types: `brain_config_change`, `brain_feedback`), `internal/chatui` (slash commands `/brain stats` and `/brain config`, next-prompt chip injector), `internal/assistant` (hook to invoke the chip generator after each answer).
- **Storage**: rolling time-series for the five metrics live alongside existing brain events; no new database. Series are bucketed daily for the last 30 days, weekly beyond that.
- **No new model**: chip generation and config-change auditing reuse qwen2.5:7b. No paid model calls.
- **Out of scope**: machine-learning the composite confidence weights from feedback events (that's a follow-up — this change just *captures* the data); a "brain dashboard" outside of chat; per-researcher quality scores (deferred until we have enough data to be meaningful).
- **Non-goals**: changing how the loop scores answers; introducing a separate analytics service; tracking user-identifiable information beyond the existing single-user assumption; supporting multi-user comparative metrics.
